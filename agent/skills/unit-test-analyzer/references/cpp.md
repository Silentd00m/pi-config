# C / C++ — Test Quality Tooling

## Coverage analysis — gcov + lcov

gcov is built into GCC. Compile with instrumentation, run tests, then collect data.

### Compile with coverage flags

```bash
# GCC
gcc -fprofile-arcs -ftest-coverage -O0 -g src/*.c -o tests_bin

# CMake project
cmake -DCMAKE_C_FLAGS="--coverage" -DCMAKE_CXX_FLAGS="--coverage" \
      -DCMAKE_BUILD_TYPE=Debug -B build
cmake --build build
```

The `-O0` is important — optimization can merge or eliminate branches, making
coverage data misleading.

### Run tests and collect

```bash
./tests_bin             # or: ctest --test-dir build/

# Collect .gcda files (written next to .gcno files after execution)
gcov src/*.c            # per-file text report
```

### lcov — aggregate and visualise

```bash
pip install lcov  # or: apt install lcov / brew install lcov

# Capture
lcov --capture --directory build/ --output-file coverage.info --rc branch_coverage=1

# Remove noise (system headers, test files themselves)
lcov --remove coverage.info '/usr/*' '*/tests/*' \
     --output-file coverage_filtered.info \
     --rc branch_coverage=1

# Text summary
lcov --summary coverage_filtered.info

# HTML report
genhtml coverage_filtered.info --branch-coverage --output-directory htmlcov/
open htmlcov/index.html
```

### Reading the HTML report

- **Lines** (blue/red): executed vs. not executed
- **Branches** (green/red arrows): `true` and `false` sides of every `if`/`?:`/`switch`
- A function shown in red with a branch arrow never taken: coverage-padding test

### Key signals

```
# Summary output:
  lines......: 94.2% (1580 of 1677 lines)
  functions..: 89.1% (82 of 92 functions)
  branches...: 61.3% (490 of 800 branches)   ← gap here is the red flag
```

A large gap between line % and branch % means tests are walking through code
without exercising its decision logic.

## Clang: llvm-cov / source-based coverage

Clang's source-based coverage is more precise than gcov — it tracks individual
expressions and conditions, not just lines.

```bash
# Compile
clang++ -fprofile-instr-generate -fcoverage-mapping -O0 -g src/*.cpp -o tests_bin

# Run
LLVM_PROFILE_FILE="tests.profraw" ./tests_bin

# Merge profiles (supports multiple test runs)
llvm-profdata merge -sparse tests.profraw -o tests.profdata

# Report
llvm-cov report ./tests_bin -instr-profile=tests.profdata    # summary
llvm-cov show   ./tests_bin -instr-profile=tests.profdata \
                -format=html -output-dir=htmlcov/            # per-line HTML
llvm-cov export ./tests_bin -instr-profile=tests.profdata \
                -format=lcov > coverage.lcov                 # lcov-compatible
```

### CMake with Clang coverage

```bash
cmake -DCMAKE_C_FLAGS="-fprofile-instr-generate -fcoverage-mapping" \
      -DCMAKE_CXX_FLAGS="-fprofile-instr-generate -fcoverage-mapping" \
      -DCMAKE_BUILD_TYPE=Debug -B build
```

## Mutation testing — mutate++ / Dextool Mutate

### mutate++ (header-only, CMake-integrated)

```bash
git clone https://github.com/nlohmann/mutate_cpp
# Follow setup: wraps your existing build + test command

# Configure in mutate++.json:
{
  "build": "cmake --build build",
  "test": "ctest --test-dir build",
  "sources": ["src/"]
}
python3 mutate++.py
```

### Dextool Mutate (more mature, CI-friendly)

```bash
# Install: https://github.com/joakim-brannstrom/dextool
dextool mutate admin --init      # creates .dextool_mutate.sqlite3
dextool mutate analyze -- -p compile_commands.json
dextool mutate test --build-cmd "cmake --build build" \
                    --test-cmd "ctest --test-dir build" \
                    --test-timeout 10s
dextool mutate report --style html --logdir report/
open report/index.html
```

Dextool requires a `compile_commands.json` (generated by CMake with
`-DCMAKE_EXPORT_COMPILE_COMMANDS=ON`).

## Combining gcov + mutation

Run gcov first to identify which branches are uncovered, then target mutation
testing at those modules. Surviving mutants in branches that *are* covered are
your worst tests — they execute the code without verifying its output.
